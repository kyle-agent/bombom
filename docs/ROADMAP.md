# bombom — Roadmap

> Living document. Captures the long-range plan and the near-term backlog.
> Last updated: 2026-06-20.

bombom의 운영 사이클은 **설계 → 확정(승인) → 릴리즈별 BOM → 현황 조회**의 반복이다.
NetBox/Nautobot의 데이터 모델을 참고하되, 우리에게 필요한 것(배치 + 원가 + 확정 워크플로우)만
얹는다. **git이 source of truth**이므로 "확정"은 별도 DB 상태가 아니라 PR 리뷰 / 머지 / 태그로
모델링한다.

```
[카탈로그] → [한 번 설계: Rack/Rack-Type 베이스라인] ─┐
                                                     ├→ [릴리즈 N: 장비 선택·배치]
                                                     │     → [확정(승인)]
                                                     │         → [투자대상 리스트 + BOM]
                                                     │             → [현황 대시보드 반영]
[릴리즈 N+1] ─────────────────────────────────────────┘  (반복)
```

핵심 매핑:

| 사용자의 요구 | 모델링 |
|---|---|
| 누가 설계하는가 | 설계자가 브랜치에서 편집 → 커밋 author = 설계자 |
| 설계 내용을 확정 | PR 리뷰 → 머지(또는 릴리즈 태그) = 확정 |
| 확정된 내용으로 BOM | 확정된 ref(main/tag) 기준으로 BOM 계산 |
| 현황 조회 | 확정 설계 롤업 대시보드 |
| Rack/Rack-Type는 한 번 설계 | 베이스라인 설계(거의 불변) |
| 매 릴리즈 장비 추가·배치·금액 | 릴리즈별 증분 placement + 투자대상 리스트 |

---

## P0 — 기반 (완료)

- 커뮤니티 카탈로그(devicetype-library) 연동, 인덱스 캐시(SQLite).
- 설계 계층 YAML: Offering → Region → Zone → Rack-Type → Rack → Device.
- BOM 엔진: pricing/ 오버레이로 CAPEX 롤업(원화).
- 웹 에디터(랙 실장도, 장비 선택·배치, 메타 입력, 저장 시 git commit), 읽기전용 뷰어.
- 가격·카테고리·메타 오버레이(카탈로그와 분리, ADR spec-cost-separation).
- 릴리즈 태그(R26.07 등), placement의 `release` 필드.
- 랙 추가 UI(POST /api/rack/new), GitHub Pages 읽기전용 export.

**Entry points:** `bombom catalog|bom|scaffold|export|serve`, `from bombom.api import create_app`.

---

## P1 — 설계 → 확정 (승인 워크플로우)  ⟵ 다음

**목표:** "누가 설계하고, 어떻게 확정하는가"를 git 흐름으로 정의한다.

- 설계자는 작업 브랜치에서 편집(에디터가 이미 커밋). 커밋 author = 설계자 신원.
- **확정 = PR 리뷰 → 머지.** 에디터에 "확정 요청" 액션 → 백엔드가 PR 생성.
- 확정 전 게이트: 필수 메타 누락·U 충돌이 0이어야 PR 생성 허용(검증은 이미 `validate_rack` 존재).
- 상태 모델: `draft`(작업중) / `in-review`(확정요청·PR open) / `confirmed`(머지됨).
- 승인 권한 분리는 P5에서 강제(여기서는 흐름만).

**산출물:** 확정 요청 API + 에디터 버튼, PR 생성 연동, 상태 뱃지, 확정 전 검증 게이트.

**왜 먼저인가:** "확정된 내용으로 BOM", "현황 조회"가 모두 *확정의 정의*에 의존한다.

---

## P2 — 릴리즈 단위 증분 설계 + 투자대상 리스트

**목표:** 베이스라인은 한 번 설계, 매 릴리즈는 추가 장비만 선택·배치·집계.

- 베이스라인(Rack/Rack-Type, 거의 불변) vs 릴리즈별 추가 placement 분리(`release` 필드 활용).
- 릴리즈 라이프사이클: 릴리즈 열기 → 장비 선택·배치 → **확정(태그 R26.xx)**.
- **투자대상 리스트 산출:** 해당 릴리즈에 추가된 placement / custom_line_item만 모아
  위치(랙 경로)·장비·수량·단가·소계·합계로 Excel/CSV 내보내기.
- 릴리즈 비교(diff): 직전 릴리즈 대비 증분(장비·금액).

**산출물:** 릴리즈 스코프 BOM 리포트(export), 릴리즈 diff 뷰, 릴리즈 열기/확정 CLI·API.

---

## P3 — 현황 대시보드 (조회)

**목표:** 확정된 설계의 현재 상태를 한눈에.

- 롤업: Offering / Region / Zone / Rack-Type별 CAPEX, 장비 수량, U 사용률.
- 릴리즈별 투자 추이(누적·증분), 상위 지출 장비, 필수 메타 미입력 현황.
- 읽기전용 뷰어를 요약 대시보드로 확장(Pages로 공유).

**산출물:** 대시보드 페이지(집계 카드 + 트리 롤업 + 릴리즈 추이), 집계 API.

---

## P4 — 가격·카탈로그 데이터 운영

- 가격북 관리: 이력, valid_from/valid_to, 벤더 견적, 통화.
- 카탈로그 서브모듈 갱신 절차, 카탈로그에 없는 **커스텀 장비** 등록 경로.
- 가격 미존재·스펙 누락 장비의 가시화/경고.

---

## P5 — 인증·권한·배포 하드닝

- 확정 권한 분리(설계자 vs 승인자), 보호 브랜치.
- 감사추적(git 이력 활용), 사내 배포(컨테이너), 백업.

---

## 시퀀싱

P1 → P2 → P3가 주 경로(확정 정의가 BOM/현황을 푼다). P4/P5는 병행 가능.
각 단계 착수 전 `/brief`로 Scope OUT 잠그고, 비자명한 선택은 `/adr`로 기록한다.
