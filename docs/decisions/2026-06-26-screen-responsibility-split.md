# ADR: 화면 책임 분리 — 기준정보 / 랙관리 / 배치, 그리고 메인 대시보드 구조

- **Status:** Accepted
- **Date:** 2026-06-26
- **Supersedes parts of:** `2026-06-24-overview-flow.md`, `2026-06-24-consolidation.md`

## Context

사용자 피드백: "랙관리 화면이 이상하다 — 랙타입과 그에 따른 랙 선택/관리만 해야 한다.
서버 배치는 이미 다른 화면(배치)에서 하고 있다. 변경비교·검증·검색은 메뉴에서 빼라."

기존 `/edit`(랙관리)는 좌측 트리 + 모델 팔레트 + 실장도 편집 캔버스 + 장비 테이블 + 확정
워크플로우까지 한 화면에 모두 담은 ~1,400줄짜리 거대 화면이었다. `/place`(배치)와 기능이
겹쳐(둘 다 장비를 랙에 배치) 사용자가 어디서 무엇을 하는지 혼란스러웠다.

## Decision

화면별 책임을 **계층 한 단계씩** 나눈다.

| 화면 | 책임 | 다루는 단위 |
|------|------|------------|
| **기준정보 `/manage`** | 조직 계층(오퍼링·리전·존·랙타입) 골격 정의 | 노드(빈 노드 포함) |
| **랙관리 `/edit`** | 랙타입 안의 **랙** 추가·복제·모델 선택 | 랙(엔클로저) |
| **배치 `/place`** | 랙에 **장비** 배치(후보풀 → 배치) | 장비 슬롯 |
| **배치 목록 `/placed`** | 배치 결과 목록 + 릴리즈 태깅 | 배치 라인 |
| **투자 리포트 `/summary`** | 집계 리포트(릴리즈 필터) | 롤업 |

- `/edit`는 실장도 편집 캔버스·장비 팔레트·확정 모달을 **전부 제거**. 랙타입을 좌측에서 고르면
  그 안의 랙들을 카드(미니 실장도 썸네일 + 모델 + U)로 보여주고, `＋ 랙 추가`(모델 선택),
  `⧉ 복제`/`⧉ 여러 개`(범위 패턴 `R10..12`), 각 랙의 `🗺 배치`(→`/place`)만 제공.
- `/manage`는 상단에 카운트 스트립(오퍼링/리전/존/랙타입/랙)과 역할 분리 안내 배너 추가.
- 백엔드 API(`/api/rack/new|clone|clone-bulk`, `/api/hierarchy`, `/api/layout`)는 그대로
  재사용 — 화면만 재구성.

### 메인 대시보드 구조

- 첫 섹션: **오퍼링 → 리전 → 존** 구조를 그대로 노출(빈 노드도 표시). 존 칩 클릭 → 배치 화면.
- 그 아래: **랙타입별 / 리전별 / 오퍼링별 투자** 막대.
- KPI는 **전체 랙·서버·리전·존 수**만. 총액 검은 배경 히어로는 제거.

### 메뉴 축소

캐노니컬 내비: `메인 · 후보풀 · 배치 목록 · 투자 리포트 · 기준정보 · 랙관리`.
변경비교(`/diff`)·검증(`/health`)·검색(`/search`)은 메뉴에서 제거. 백엔드 라우트/모듈은
유지(테스트·재사용 가능) 하되 데모 내비에서는 노출하지 않는다.

## Consequences

- `/edit`와 `/place`의 기능 중복이 사라지고, 각 화면이 계층 한 단계만 책임진다.
- `build_overview`는 디렉터리 계층(`list_hierarchy`)으로 트리를 만들되 **워크스페이스 루트**가
  필요하다. API가 노드 경로(`offerings/...`)를 넘기므로, 트리는 `ws.root`에서, 배치 집계는
  넘어온 노드 경로에서 각각 읽도록 분리했다(이전엔 같은 인자를 둘 다에 써서 트리가 비었음).

## Alternatives considered

- **diff/health/search 기능 완전 삭제(라우트·모듈·테스트까지):** 보류. 릴리즈 비교·검증은
  실제 기능이고 테스트가 있어, 메뉴 비노출로 충분. 완전 삭제는 사용자 확인 후 별도 작업.
