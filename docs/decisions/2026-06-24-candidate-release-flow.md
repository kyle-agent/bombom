# 2026-06-24 — 후보풀 중심 흐름 + 릴리즈 태깅 분리 (IA/CX 재정리)

## Context
사용자가 의도한 제품 흐름: 커뮤니티 카탈로그에서 **1차 후보**를 뽑고(가격·비고 입력), 후보는
랙 배치 여부와 무관하게 항상 후보 목록에 보이며, 배치 화면에서도 무조건 후보로 노출된다.
새로 배치한 장비를 **어느 릴리즈로 구분할지는 배치 화면에서 정하면 복잡**하므로, 배치된 장비
목록 화면에서 **태그 붙이듯 따로** 정한다. 리포트는 특정 릴리즈만 골라 뽑을 수 있어야 한다.

## Decision
엔티티: **카탈로그 → 1차 후보(Candidate Pool) → 배치(Placement) → 릴리즈 태그 → 리포트**.

- **후보풀이 마스터.** `/place`의 배치 소스 팔레트 = 전체 후보풀(항상 노출, 검색 필터만).
  ＋담기식 큐레이션 제거 — 후보는 `/candidates`에서 관리, 배치 화면은 그걸 그대로 소비.
- **릴리즈는 배치에서 분리.** 새 배치 = **미태그(`release: DRAFT`)**. `/place`에 릴리즈 선택 없음.
- **신규 화면 `/placed` (배치 목록·릴리즈 태깅).** 랙별 배치 장비 목록에서 장비를 골라 릴리즈
  태그 부여/변경(미태그→릴리즈, 새 태그 입력 가능). 저장은 랙별 `PUT /api/rack`(릴리즈 문자열만
  바뀜 → 구조 검증 통과). 릴리즈 드롭다운은 전체(global) 릴리즈 노출.
- **리포트 릴리즈 필터.** `build_overview(release=...)` + `/api/overview?release=`, 응답에
  `releases` 목록 추가. `/summary`에 전체/각 릴리즈/미태그 필터.

## Consequences
- 흐름이 한 줄로 명확: 후보 정의 → 배치 → 태깅 → 리포트. 배치와 릴리즈 결정이 분리돼 배치 화면이
  단순해짐.
- `DRAFT`는 자유 문자열 릴리즈값(작업본). 메인/리포트 무필터엔 포함, 특정 릴리즈 필터엔 제외.
  `/place`에서 점선 테두리로 미태그 표시.
- **Scope OUT(후속):** 랙에 안 들어가는 "사용 장비"(custom_line_items)를 후보→사용으로 직접
  올리는 UI, 태깅 화면에서의 메타(시리얼 등) 일괄 입력, 리포트 경로 스코프 필터.
