# 2026-06-20 — 복제(clone): 랙 단위 + 서브트리 단위, working-tree 복사·확정 무관

## Context
디자이너의 핵심 작업은 **비슷한 하드웨어를 다수 배치**하는 것이다(존 하나에 동일 랙 N개, "new
build"로 기존 리전/존을 통째로 복제 후 수정). 지금까지 랙은 매번 백지에서 짜야 했고, 서브트리
복제 헬퍼(`scaffold.clone_subtree`)는 CLI에만 연결돼 화면에서 쓸 수 없었다. locked UI에도
"region/zone clone for new builds"가 명시돼 있다.

## Decision
복제를 두 단위로 나누고 **둘 다 working-tree 복사**로 둔다(확정 태그/릴리즈 의미론과 무관).

- **랙 복제** (`POST /api/rack/clone`, `scaffold.clone_rack`): 한 랙 YAML을 **같은 Rack-Type
  디렉터리** 안 새 rack id로 byte-for-byte 복사한다. 모든 Placement(device/position/release/
  qty/meta)와 custom line item이 그대로 보존된다 → "랙 하나 짜고 똑같이 N개"가 호출 N번.
  복사본은 목적지 경로에서 계층을 상속하고, 커밋 전에 `load_racks`+`validate_rack`로 검증한다
  (실패 시 복사 파일 unlink 후 400). 대상 id가 이미 있으면 409.
- **서브트리 복제** (`POST /api/hierarchy/clone`, 기존 `clone_subtree` 노출): 존/Rack-Type/
  리전/오퍼링 노드를 **형제로** 복제한다. 하위 Rack-Type·랙(배치 포함) 전체가 `copytree`로
  복사되고, 노드 자신의 meta `name`만 새 이름으로 바뀐다. 형제 충돌 시 409.
- **범위 한정(v1)**: 랙 복제는 같은 Rack-Type 안에서만(다른 존/타입으로의 이동 복제는 서브트리
  복제가 담당). 한 번에 N개 일괄 복제·복제 시 release 재태깅은 Scope OUT(UI가 반복 호출로 처리).
- **id 안전성**: 새 id/이름은 `_safe_id`(`[A-Za-z0-9][A-Za-z0-9._-]*`, `..` 금지)로 모델·
  엔드포인트 양쪽 검증, `clone_rack`은 라이브러리 호출에도 자기방어 가드를 둔다. 소스 경로는
  `_resolve_rack_write`/`node_dir`로 워크스페이스 밖을 거부한다.

## Consequences
- 신규: `scaffold.clone_rack`, API `POST /api/rack/clone`·`POST /api/hierarchy/clone`,
  에디터 헤더 `⧉ 복제`(loadRack로 복제본 오픈), `/manage` 노드별 `⧉ 복제`.
- 확정(annotated tag) 스냅샷은 불변 이력이라 복제와 무관하다 — 복제는 순수 working-tree 편집.
- 테스트: 랙 복제 placements 보존·409·unsafe-id, 서브트리 복제·placements 보존·409(무커밋)·
  unsafe-name. 102 pass, ruff clean. 보안 리뷰 PASS(traversal·overwrite 가드 확인).

## Override conditions
존/타입을 넘나드는 랙 복제나 일괄 N개 복제 요구가 커지면 목적지 경로를 받는 형태로 랙 복제 API를
넓힌다. 복제 시 release를 다음 빌드로 자동 승급해야 하면(누적 빌드 모델이 확정되면) 복제 단계에서
release 재태깅 옵션을 추가한다 — 이는 decommission/release 의미론 결정과 함께 다룬다.
