# 2026-06-19 — App stack: FastAPI backend + JS frontend, server-rendered SVG + form editing

## Context
Designers need a NetBox-like rack elevation view on screen plus a way to select HW, set
quantity, and place it — with all edits reflected to git. This adds a frontend on top of
the git-backed data + BOM core.

## Decision
- **Backend: Python / FastAPI.** Owns the git datastore (read working copy, commit on
  write), catalog sync, rebuildable SQLite index, BOM engine, and **server-side rack
  elevation SVG rendering** (computed from rack u_height + device position/u_height/face,
  NetBox-style). Exposes a REST API.
- **Frontend: JS SPA (React or Svelte — TBD).** Hierarchy tree, elevation view, HW picker
  with **form-based** placement, live BOM/CAPEX panel.
- **Edit loop:** form submit → API → write rack YAML → git commit. No interactive
  drag-and-drop in MVP.
- **Cost scope: CAPEX-first.** Power rolled up as capacity (W); cost conversion deferred.

## Alternatives considered
- **Full Python server-rendered (Django/Jinja + HTMX).** Simplest, mirrors NetBox, but the
  team chose a dedicated JS frontend for richer interaction headroom.
- **Full TypeScript (Next.js).** One ecosystem, but the catalog YAML/schema tooling and
  BOM math are stronger and already Python-aligned (devicetype-library + import tool).

## Consequences
- Two languages to build/operate; a clear REST API contract between them.
- Python keeps catalog/BOM/git logic close to the upstream tooling and schema.
- Server-rendered SVG means elevation logic is testable backend code, not frontend-only.

## Override conditions
If the form-based edit loop proves too limiting, upgrade to interactive editing (separate
decision). If the JS frontend overhead isn't justified, fall back to server-rendered + HTMX.
